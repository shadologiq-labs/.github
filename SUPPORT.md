# Support

ShadoLogiq Labs is run by users, for users. There's no support contract, no SLA, no help desk. What we have is GitHub issues and a bias for fixing things.

---

## Before opening an issue

1. **Check the project's README** — most "how do I..." questions are answered there
2. **Search existing issues** — yours might already be filed (or closed)
3. **Skim the `CLAUDE.md`** — architecture decisions and "why we do it this way" live there
4. **Try the latest version** — bugs are often already fixed on `main`

If you've done those four and still need help, file an issue. Pick the project repo, not this one.

---

## Bug reports

Open a **Bug report** issue. The template asks for:

- **What happened** vs **what you expected**
- **Steps to reproduce** — exact, copy-pasteable where possible
- **Environment** — browser + version, OS, the platform you're using it on (e.g., specific ServiceNow instance type), and the project version
- **Screenshots** if visual
- Sanitize anything sensitive before posting

The more reproducible the bug, the faster it gets fixed.

## Feature requests

Open a **Feature request** issue. Tell us:

- **The user pain** — what default behavior is bothering you
- **What you'd want instead** — your proposed solution
- **What you're doing today** — current workaround, if any

We're open to most things that fit a project's scope. We won't add features that drift the project away from its core purpose, that require a server, or that introduce telemetry.

## Questions

If it's not a bug or a feature request — say, "how do I configure X?" or "is this approach reasonable?" — open an issue with the `question` label, or use Discussions if the repo has them enabled.

---

## Security issues

**Don't** open a public issue. See [SECURITY.md](./SECURITY.md) for the responsible disclosure process.

---

## Response time

We're volunteers. Reasonable targets:

- **Acknowledge** within a few days
- **Triage** within a week
- **Fix critical bugs** as fast as we can, prioritized by impact

If something is urgent, say so in the issue and explain why. We'll triage accordingly.

---

## What we won't help with

- Configuring third-party tools we don't own (e.g., "how do I make ServiceNow do X" — that's ServiceNow's problem, not the SNapp Extension's)
- Custom integration work for individual deployments
- Feature requests that violate the project's stance (telemetry, monetization, dependency bloat)
